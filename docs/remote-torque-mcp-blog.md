# One Torque Talking To Another

Agent infrastructure is most useful when it can do real work without becoming a privileged shell with better autocomplete. That was the design line for Torque's MCP server. The server exposes build, ship, verify, stack, logs, capture, and session tools through structured MCP calls, while the actual heavy work still runs through Torque's existing CLI and gRPC agent boundary.

The interesting part is the bridge. One Torque process can run as an MCP server for an agent host, receive a request such as "build this image and ship the stack", and forward that operation to another Torque endpoint over gRPC. The agent sees typed tools like `torque.build.run`, `torque.ship.run`, and `torque.stack.apply`. The remote side sees the same production services it already trusts: `BuildService.RunBuild`, `DeployService.Apply`, `StackService.Apply`, `MirrorService.Subscribe`, and the rest of the agent API.

That separation matters. MCP is the agent contract. gRPC is the execution contract. The MCP server does not reinvent deploy logic, stream parsing, stack ordering, or remote session storage. It translates requests, enforces policy, redacts secrets, and returns evidence the model can inspect. Mutating calls require server-side write enablement and request-level confirmation, so a model cannot accidentally turn a read-only review into a cluster write. Remote tokens travel through environment variables instead of command arguments, and session output is scrubbed before it is returned to the caller.

To make this more than a happy-path demo, the repo now includes a scenario catalog with sixteen complex DAG and build cases. They cover fanout image builds, diamond compose promotion, large layered stack runs, sealed review before apply, multi-namespace delete thresholds, secret-provider builds, git-range affected releases, canary diagnosis, failed-node resume, policy-blocked builds, cross-cluster read-only plans, HTTP MCP origin checks, mirror replay, microVM smoke tests, verifier fan-in, and rollback delete after failed ship. Each scenario records the topology, build mode, MCP calls, remote gRPC services, safety expectations, and validation signals.

The catalog is intentionally executable in spirit. A regression test checks that the file stays between ten and twenty scenarios, that every topology is a valid acyclic graph, that edge endpoints exist, and that required surfaces such as `torque.ship.run`, `torque.stack.delete`, remote StackService, MirrorService, write confirmation, and remote token redaction remain represented. This keeps the scenarios useful for future agent evaluations instead of letting them drift into stale documentation.

The end-to-end tests exercise the bridge directly. A fake remote Torque agent receives MCP-triggered gRPC calls for build, apply, delete, verify, logs, and stack operations. Another test proves the build-to-ship flow: MCP calls `torque.build.run`, then launches `torque ship` with a remote agent target, while ensuring the remote token never appears in argv, the MCP response, or the session tail. On the Selectel lab host, the remote gRPC and MCP suites were run 100 times, and a static `torque-mcp` binary was booted inside QEMU microVMs 100 times to verify initialize, tool listing, info, ship, and stack tool advertisement.

The result is a practical pattern for agents: expose a small, typed, auditable MCP surface, keep execution behind hardened Torque services, and make every build or deploy leave evidence that another process can review. Agents get useful power, operators keep control, and remote delivery stays inspectable from first plan to final stack status.
