Pair ktl stack Concurrency with apply plan --visualize

11 min read Updated February 24, 2026

Use a two-layer deploy model: visualize release-level blast radius first, then execute dependency-safe concurrent rollouts across the stack.

Published February 24, 2026

Parallel deploys are only useful if operators can still answer two questions quickly: What is changing, and why did rollout behavior change? The strongest pattern in ktl is combining ktl apply plan --visualize with ktl stack DAG concurrency.

The Two-Layer Model

Layer 1 is release-level visualization. Layer 2 is stack-level execution. You use both in the same runbook:

  • First, generate HTML plan views for critical releases and share them in PR or CI artifacts.
  • Then execute the full stack with DAG-aware concurrency and adaptive backoff controls.
# Layer 1: review the change surface
ktl apply plan \
  --chart ./charts/api \
  --release api \
  --namespace platform \
  -f ./values/prod/api.yaml \
  --visualize \
  --output ./artifacts/plan-api.html

# Optional regression guard against a previous plan
ktl apply plan \
  --chart ./charts/api \
  --release api \
  --namespace platform \
  -f ./values/prod/api.yaml \
  --compare-to ./baselines/api-plan.json \
  --baseline ./baselines/api-plan.new.json

# Layer 2: run the whole graph
ktl stack apply --config ./stacks/prod --yes

Why This Beats Plain Parallel Flags

A fixed parallel number alone is not enough in real clusters. ktl stack executes only ready nodes in DAG order, and can adjust worker target when failures indicate pressure (rate limits, transport errors, 5xx behavior).

Pairing this with plan visualization means reviewers see expected resource changes before rollout, while operators keep high throughput during execution.

Recommended Runner Guardrails

Keep concurrency controls in stack.yaml so behavior is consistent between laptop and CI: 

apiVersion: ktl.dev/v1
kind: Stack
name: prod

runner:
  concurrency: 8
  progressiveConcurrency: true
  limits:
    maxParallelPerNamespace: 2
    maxParallelKind:
      StatefulSet: 1
      Job: 2
    parallelismGroupLimit: 2
  adaptive:
    mode: balanced
    min: 1
    window: 20
    rampAfterSuccesses: 2
    rampMaxFailureRate: 0.30
    cooldownSevere: 4

releases:
  - name: api
    chart: ./charts/api
  - name: worker
    chart: ./charts/worker
    needs: [api]

Comparison: Helmfile and Centralized CI/CD

Concern ktl model Helmfile-style orchestration Centralized CI/CD only
Concurrency behavior DAG-safe plus adaptive controls and scope limits. Usually fixed parallelism, less runtime adaptation. Parallelism mostly at pipeline job/stage level.
Pre-apply review quality apply plan --visualize gives shareable interactive diff/graph views. Depends on external tooling/scripts. Depends on platform dashboards and custom jobs.
Failure recovery Resume/rerun-failed workflows with stack state context. Often manual rerun selection. Often replay entire pipeline stages.
Operator feedback loop Local and CI parity with stack console and event stream. CLI output only, less integrated runtime context. Good governance, slower interactive debugging loop.

Pipeline Pattern That Scales

A practical sequence for teams with approval gates:

  • PR stage: run ktl apply plan --visualize for changed releases, publish HTML artifacts.
  • Approval stage: reviewers inspect rendered diffs and comparison deltas.
  • Deploy stage: run ktl stack apply with config-defined concurrency guardrails.
  • Incident stage: run ktl stack apply --resume --yes after fixing root cause.
# after fixing a failed release
ktl stack apply --config ./stacks/prod --resume --yes

# follow run status
ktl stack status --config ./stacks/prod --follow

Final Take

If your deploy model needs both speed and confidence, the combination is straightforward: apply plan --visualize for what will change, and stack DAG concurrency for how rollout executes under load. This gives a better balance than either fixed parallel orchestration or fully centralized pipeline-only execution.

Suggested starting point: set runner.concurrency to 4-8, turn on progressiveConcurrency, and cap StatefulSet parallelism to 1.
