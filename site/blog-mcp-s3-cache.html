One Torque Talking To Another

A practical pattern for giving agents typed delivery tools without turning them into privileged shells.

Torque exists because not every delivery loop belongs in a hosted CI control plane. Some teams ship from air-gapped networks, regulated enclaves, factory floors, field clusters, sovereign clouds, or short-lived labs where runners cannot phone home and artifacts must stay local. Others need a portable CI-in-a-box workflow for disaster recovery, customer reproductions, edge upgrades, or build-and-deploy evidence that can move with a case file. In those places, the hard problem is not starting a pipeline; it is keeping the pipeline inspectable, replayable, and narrow enough to trust. Torque treats the workstation, VM, or isolated build host as the delivery cell: build, verify, plan, apply, capture, and explain all stay explicit. That gives agent-driven CI/CD a safer substrate: agents can request work through typed tools while operators keep the execution boundary, evidence, and approvals under control.

Agent infrastructure is most useful when it can do real work without becoming a privileged shell with better autocomplete. That was the design line for Torque's MCP server. The server exposes build, ship, verify, stack, logs, capture, and session tools through structured MCP calls, while the actual heavy work still runs through Torque's existing CLI and gRPC agent boundary.

The interesting part is the bridge. One Torque process can run as an MCP server for an agent host, receive a request such as "build this image and ship the stack", and forward that operation to another Torque endpoint over gRPC. The agent sees typed tools like torque.build.run, torque.ship.run, and torque.stack.apply. The remote side sees the same production services it already trusts: BuildService.RunBuild, DeployService.Apply, StackService.Apply, MirrorService.Subscribe, and the rest of the agent API.

That separation matters. MCP is the agent contract. gRPC is the execution contract. The MCP server does not reinvent deploy logic, stream parsing, stack ordering, or remote session storage. It translates requests, enforces policy, redacts secrets, and returns evidence the model can inspect. Mutating calls require server-side write enablement and request-level confirmation, so a model cannot accidentally turn a read-only review into a cluster write. Remote tokens travel through environment variables instead of command arguments, and session output is scrubbed before it is returned to the caller.

That is also the answer to "how do you make a secure MCP server?" Keep MCP boring and narrow. Prefer stdio for local agent hosts. If HTTP is needed, bind it to loopback, require bearer auth, validate browser origins, cap request sizes, and put any public access behind a real TLS proxy. Do not let MCP become a generic shell or a direct Kubernetes tunnel. It should be a typed contract with small tools, bounded inputs, redacted outputs, and clear write gates.

The remote side should be harder than the MCP side. A production torque-agent runs behind TLS and client certificate verification, keeps the bearer token as a second factor, uses a scoped kubeconfig and Kubernetes RBAC, writes MirrorService sessions to durable storage, and puts retention limits on that evidence. MCP write tools still need server enablement plus per-request confirmation, and every mutating action has to leave a capture, plan digest, verifier report, ship summary, stack run ID, or mirror replay behind.

The target posture is simple: MCP over stdio or authenticated loopback HTTP, remote gRPC over mTLS, writes disabled unless explicitly enabled, secrets never crossing MCP, and every action replayable from evidence.

That posture is now installable as a Linux service instead of a hand-written runbook. install.sh --mode systemd-daemon installs torque, torque-agent, and torque-mcp, writes /etc/torque/agent.env with separate gRPC and MCP bearer tokens, and starts torque-agent.service plus torque-mcp.service. The agent runs in durable mode, persists MirrorService frames under /var/lib/torque/agent/mirror.sqlite, and requires sandbox execution for remote builds. The MCP service listens on 127.0.0.1:7331, requires bearer auth, and bridges to the local gRPC agent at 127.0.0.1:7443.

The service path was smoke-tested in two systemd environments: Ubuntu 24.04 and a nested Fedora 42 Cloud microVM. In both cases the installer wrote the units, restarted already-running services during upgrade, kept bearer tokens out of process arguments, created the durable mirror database, returned 401 for unauthenticated MCP calls, accepted authenticated tools/list, and served torque.info through the MCP-to-gRPC bridge. systemd-analyze verify accepted both unit files.

The cache path follows the same rule. Instead of asking an agent to read BuildKit logs and guess whether a layer was reused, Torque exposes torque.cache.inspect, torque.cache.plan, and torque.cache.warm. Inspect returns normalized cacheFrom and cacheTo entries, the S3 cache manifest name, warnings, and cache-intel evidence when a local cache directory is available. Plan accepts changed paths and classifies them as Dockerfile, dependency-layer, source-layer, deploy-input, or CI-input changes. Warm is the only mutating step: it requires --enable-write and safety.confirm=true, then calls remote BuildService.RunBuild with push, load, and signing disabled while cache exports stay enabled.

To make this more than a happy-path demo, the repo now includes a scenario catalog with sixteen complex DAG and build cases. They cover fanout image builds, diamond compose promotion, large layered stack runs, sealed review before apply, multi-namespace delete thresholds, secret-provider builds, git-range affected releases, canary diagnosis, failed-node resume, policy-blocked builds, cross-cluster read-only plans, HTTP MCP origin checks, mirror replay, microVM smoke tests, verifier fan-in, and rollback delete after failed ship. Each scenario records the topology, build mode, MCP calls, remote gRPC services, safety expectations, and validation signals.

The catalog is intentionally executable in spirit. A regression test checks that the file stays between ten and twenty scenarios, that every topology is a valid acyclic graph, that edge endpoints exist, and that required surfaces such as torque.cache.plan, torque.cache.warm, torque.ship.run, torque.stack.delete, remote StackService, MirrorService, write confirmation, and remote token redaction remain represented. This keeps the scenarios useful for future agent evaluations instead of letting them drift into stale documentation.

The end-to-end tests exercise the bridge directly. A fake remote Torque agent receives MCP-triggered gRPC calls for build, apply, delete, verify, logs, and stack operations. Another test proves the build-to-ship flow: MCP calls torque.build.run, then launches torque ship with a remote agent target, while ensuring the remote token never appears in argv, the MCP response, or the session tail. On a Linux lab host, the remote gRPC and MCP suites were run 100 times, and a static torque-mcp binary was booted inside QEMU microVMs 100 times to verify initialize, tool listing, info, ship, and stack tool advertisement.

The result is a practical pattern for agents: expose a small, typed, auditable MCP surface, keep execution behind hardened Torque services, and make every build or deploy leave evidence that another process can review. Agents get useful power, operators keep control, and remote delivery stays inspectable from first plan to final stack status.
